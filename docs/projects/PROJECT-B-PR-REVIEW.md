# Project B: Translation PR Review & Quality Feedback

> **Type**: QA-focused, bilingual review  
> **Ideal RA profile**: Strong bilingual fluency (English + Chinese), attention to detail, interest in translation quality  
> **Time**: 4–7 hrs/week for 12 weeks  
> **Repository**: Works in `QuantEcon/test-translation-sync.zh-cn` (existing) and documents findings in `QuantEcon/project-translation-benchmark`

## Goal

Systematically review the translation PRs generated by `action-translation`, document error patterns, and produce a structured set of improvement recommendations. Your findings directly drive quality improvements in the translation tool.

## Why This Matters

`action-translation` generates translation PRs automatically, but we have limited data on **where it fails and how**. We need someone with bilingual fluency and domain knowledge to carefully review real outputs and tell us:

- What kinds of errors does the tool make?
- How often does it get technical terms wrong?
- Does it preserve formatting correctly?
- Are there systematic patterns we can fix?

This is the feedback loop that makes the tool better.

## What You'll Do

### 1. Learn the Test Workflow

The `action-translation` repo includes a test tool that generates PRs across 24 different translation scenarios. You'll use this to create a controlled set of PRs to review.

**Test repositories**:

| Repo | Role |
|------|------|
| `QuantEcon/test-translation-sync` | English source content |
| `QuantEcon/test-translation-sync.zh-cn` | Chinese translations (PRs appear here) |

**Test scenarios** cover situations like:
- New section added to a document
- Existing section modified
- Section deleted
- Mathematical content changed
- Code cell comments updated
- Deep subsection nesting (####, #####)
- Multiple changes in one PR

### 2. Review Translation PRs

For each PR in the target repository, you'll assess:

| Dimension | What to Look For |
|-----------|-----------------|
| **Accuracy** | Does the translation convey the same meaning as the English? |
| **Terminology** | Are technical terms translated consistently and correctly? (Check against `glossary/zh-cn.json`) |
| **Fluency** | Does the Chinese read naturally? Is it the right register for a university textbook? |
| **Formatting** | Are code blocks, math equations, MyST directives, and links preserved? |
| **Heading-maps** | Are the `heading-map` entries in the frontmatter correct? |
| **Scope** | Did the tool translate only the changed sections, or did it touch things it shouldn't have? |
| **Structure** | Are sections in the right order? Any duplicated or dropped content? |

### 3. Build an Error Taxonomy

As you review PRs, categorize the issues you find. Start with these categories and refine as you go:

| Category | Example |
|----------|---------|
| **Term mistranslation** | "Bellman equation" translated inconsistently across sections |
| **Register error** | Translation too casual or too formal for textbook context |
| **Formatting broken** | Code block lost its language tag, math delimiter changed |
| **Section duplicated** | Same content appears twice after translation |
| **Section dropped** | Content from the source is missing in the translation |
| **Heading-map wrong** | Heading ID doesn't match the translated heading text |
| **Over-translation** | Tool translated content it shouldn't have (e.g., code, URLs) |
| **Under-translation** | English text left untranslated in the Chinese output |
| **Math error** | LaTeX modified or broken during translation |
| **Scope creep** | Tool modified sections that weren't changed in the source |

### 4. Produce a Structured Report

For each PR reviewed, record:

| Field | Description |
|-------|-------------|
| **PR number** | Link to the PR in the target repo |
| **Test scenario** | Which of the 24 scenarios this PR tests |
| **Overall quality** | 1–10 score |
| **Issues found** | List of issues with category, severity (minor/major/critical), and specific example |
| **Suggested fix** | What the correct translation should be, or what the tool should have done differently |

Compile these into a summary report with:
- Frequency of each error category
- Most common failure patterns
- Prioritised list of improvements (what would fix the most errors?)

### 5. Review Real Lecture Translations

Once comfortable with the test scenarios, apply the same review process to translation PRs from real QuantEcon lecture content (if available). These are more complex and will surface issues that the controlled test scenarios don't cover.

## Week-by-Week Plan

### Weeks 1–2: Setup & Orientation

- [ ] Read the `action-translation` [README](../../README.md) and [architecture docs](../developer/architecture.md)
- [ ] Understand how the test tool works: read `tool-test-action-on-github/README.md`
- [ ] Get access to the test repositories
- [ ] Review your first 2–3 PRs to develop your workflow
- [ ] Draft initial review template (spreadsheet or structured markdown)

### Weeks 3–5: Systematic Test PR Review

- [ ] Review all 24 test scenario PRs
- [ ] Record issues in structured format
- [ ] Start building the error taxonomy
- [ ] Flag any critical bugs found (open issues in `action-translation`)

### Weeks 6–8: Error Taxonomy & Patterns

- [ ] Refine error taxonomy based on all reviews
- [ ] Identify the top 5 most frequent error patterns
- [ ] Cross-reference term errors against `glossary/zh-cn.json` — are missing terms the cause?
- [ ] Write up pattern analysis

### Weeks 9–10: Real Lecture PRs

- [ ] Review translation PRs from real QuantEcon lecture content
- [ ] Compare error patterns: do real lectures show the same issues as test scenarios?
- [ ] Note any new error categories that emerge

### Weeks 11–12: Summary & Recommendations

- [ ] Write final summary report
- [ ] Produce a prioritised list of improvement recommendations
- [ ] Identify terms that should be added to the glossary (hand off to Project C)
- [ ] Identify test cases that should be added to the benchmark dataset (hand off to Project A)

## Deliverables

1. **Structured review of all test PRs** — one entry per PR with scores and issues
2. **Error taxonomy** — a categorised, documented list of error types with examples
3. **Pattern analysis** — which errors are most common, which are most severe
4. **Improvement recommendations** — prioritised list of what to fix, with specific examples
5. **Cross-project handoffs** — missing terms for Project C, test cases for Project A

## Tools You'll Use

- **GitHub**: Reviewing PR diffs, leaving comments, opening issues
- **`glossary/zh-cn.json`**: Reference for expected term translations (357 terms)
- **`tool-test-action-on-github/`**: Generating test PRs
- **Spreadsheet or Markdown**: Recording structured review data
